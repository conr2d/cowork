# WP6 — Full gate runbook (v0.2 definition of done)

The **authoritative** v0.2 acceptance gate. Like the v0.1 gate it runs **manually /
semi-automated** on a real Windows clean-room; everything cheaply automatable already runs
per-push in `ci.yml` (frontend, guest/host unit tests, conformance, docker `guest-e2e`,
real-WSL2 `wsl-integration`, and the windows-latest `host` job that compiles `src-tauri`).
This gate covers what those cannot: the real ConPTY multi-session lifecycle, real agent
resume across app restarts, and real first-run sign-in flows.

**Clean-room mechanics (VM creation, checkpoints, baseline verification, i18n passes) are
identical to v0.1 — follow [`wp10-full-gate.md`](./wp10-full-gate.md) §1–§3 and reuse
`scripts/wp10/fullgate.ps1`.** This document only defines what v0.2 adds or changes.

---

## 0. The freeze rule — how v0.2 actually closes

**Read this before starting a run.**

The first attempt at this gate never finished. Each defect found in the field was fixed on the
spot, which invalidated the installer, which restarted the run: reboot-resume gap → fix → build
→ truncated sign-in URL → fix → build → and so on. The artifact never stopped moving, so no run
ever completed. Without a freeze, v0.2 does not close — not because the product is bad, but
because the process has no terminating condition.

So:

1. **Freeze.** Cut one release artifact. From that point, `main` may keep moving but **the gate
   runs against that one artifact**.
2. **Run it end to end.** Defects found mid-run are **recorded, not fixed.** Finish the run.
3. **Triage the log against the bug bar below.** Fix only what blocks. Everything else goes to
   the `v0.2-polish` bucket and is *not* a reason to re-cut.
4. **Re-cut and re-run only if a blocker was fixed.** Otherwise v0.2 is closed.

### Bug bar

| Blocks the close | Does not block |
|---|---|
| A non-developer **cannot complete setup** | Setup completes, but it is rough |
| The user **loses work**, or data is corrupted (e.g. two instances racing on one distro) | Cosmetic, layout, wording |
| A **release-blocking section** below fails | Anything that needs the scheduled design overhaul |
| An agent **cannot be signed into at all** | Anything only a developer would notice, or that the UI already offers a way around |

Judged by this bar, the sign-in URL truncation and the missing single-instance guard were
blockers. The lingering sign-in button, the unannounced UAC prompt, and the frozen-looking long
steps were **not** — they were fixed anyway, on a separate track, because design is a real moat
for this product. Keep those two tracks separate: *quality work does not gate the close, and the
close does not wait on quality work.*

**Closing v0.2 means:** one artifact, one complete run, zero release-blocking failures, log
appended to §10. It does not mean a public release — nothing ships publicly at v0.2.

Scope sealed on `main` as of `0f03e5f`: WP0–WP5 + boot-time guest re-injection
(multi-session PTY `7d08d9b`, session tabs + claude resume `9184f48`, codex/agy
continuity `479ee4d`, presets `37b1bf8`, guest re-injection whenever the installed bytes
differ from the shipped ones `2f9a6d1`), plus the changes made after that scope was first
written:

- `816317a` — **no eager auth.** Cowork never runs an agent's login command; each agent
  drives its own sign-in on first launch. (§3, §7)
- `1e9fe85` — toolchain activation moved to `~/.profile`, so every *login* shell has brew
  and mise on `PATH`. Verified on the real WSL2 kernel by CI's `wsl-integration` job; **not
  a gate item.**
- `eab7522` — **credentials are no longer routed.** Each agent uses its own default config
  directory; nothing is created under `~/.cowork`. (§1, §6, §7)
- `0f03e5f` — **`agent-install` resolves before installing.** An agent already on a login
  shell's `PATH` is used as-is, never reinstalled. (§3b)
- `dc912ea` — upgrade/back-compat affordances dropped. A build assumes a **fresh install**;
  the setup marker moved to `%LOCALAPPDATA%\Cowork\setup-complete` (extensionless, empty), so
  a machine carrying the old `setup-complete.json` re-runs the wizard. Expected.
- `dc262f0` — **the reboot-resume gap.** Resume is keyed on `resume.json`, not the `--resume`
  argv, so a manual launch resumes correctly; a single-instance guard prevents a second
  window; the boot gate paints a splash. (§3a — new, and it supersedes wp10 §4.5)

### 3a. Reboot + resume (`dc262f0`)

RunOnce is executed by Explorer, which defers startup items until the desktop goes idle. The
relaunch is therefore **late** — that is Windows, not a defect. What is release-blocking is
that the user is never stranded by the delay.

| Step | Action | Expected (PASS) |
|------|--------|-----------------|
| 3a.1 | Read the pre-reboot screen | it states the delay ("up to a minute") **and** that opening Cowork yourself is fine. Not a vague "will resume automatically" |
| 3a.2 | Reboot, sign back in, **wait** without touching anything | Cowork reopens by itself and resumes at **provision** with the pre-reboot agent selection intact. Note the observed delay in the log — a long one is a PASS, a no-show is a FAIL. The `RunOnce` value is consumed (check `reg query HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce` — the `Cowork` value is gone; it must not loop on the next boot) |
| 3a.3 | **Impatient path** (revert, redo the reboot): sign in, then open Cowork yourself from the Start menu *before* it reopens on its own | the wizard resumes at **provision**, NOT at the language step; the agent selection survived. This is the point of the copy in 3a.1 |
| 3a.4 | Keep that manually-opened window in the foreground and wait for RunOnce to fire | **no second window.** The existing window is focused/unminimized instead. Task Manager shows exactly one `cowork-app.exe`. (Two instances would drive provisioning twice against one distro and race on `workspaces.json`) |
| 3a.5 | Minimize the window, then trigger a second launch (double-click the shortcut) | the minimized window is restored and focused; still one process |
| 3a.6 | Every launch, from the moment the window appears | it paints the **Cowork wordmark + spinner** immediately — never a blank white window while it decides wizard-vs-shell |

### 3b. Wizard deltas

| Step | Action | Expected (PASS) |
|------|--------|-----------------|
| 3.1 | After agent install | the wizard shows the **done step** — no terminal, no login buttons, no auto-advance. Clicking "Open Cowork" enters the shell |
| 3.2 | Relaunch after clicking **Open Cowork** | marker written; relaunch boots straight to the shell |
| 3.3 | After finish: `wsl -d Cowork -- bash -lc 'ls -d ~/.cowork'` | **no such directory** — credentials are not routed |
| 3.4 | Re-run the install by hand: `wsl -d Cowork -u cowork -- /usr/local/bin/cowork agent-install --agent claude` | returns in **seconds** with no installer output — the agent already on `PATH` is resolved, not reinstalled. `claude --version` still passes |
| 3.5 | Pre-seeded binary: from a `ready-to-install` run, stop after the distro exists, install an agent by hand into a non-default location on `PATH` (e.g. `sudo install -m0755 ~/.local/bin/claude /usr/local/bin/claude && rm ~/.local/bin/claude`), then run `cowork agent-install --agent claude` | resolves `/usr/local/bin/claude` and skips the install; **no** `~/.local/bin/claude` reappears |

## 9. Failure induction (delta)

| Case | Induce | Expected (PASS) |
|------|--------|-----------------|
| 9.1 | Corrupt the installed guest: `wsl -d Cowork -u root -- bash -c 'echo broken > /usr/local/bin/cowork'`, relaunch the app | shell boots; guest re-injected (sha matches shipped); workspace create works |
| 9.2 | Kill `wsl.exe` for a running session (Task Manager, or `Get-Process wsl` → `Stop-Process -Id <pid> -Force` — no Ctrl+Alt+Del needed) | tab flips to **done**; app stays responsive; other tabs unaffected; new sessions spawn fine |

**Dropped: the old 9.2 (`chmod -x` the guest, expect it to heal).** It defended a failure
mode nobody produces — the exec bit does not vanish on its own — and the only way to cover it
was a `wsl.exe` spawn on every shell boot, forever. Corrupt bytes and a missing binary are
covered by 9.1, which is what actually happens. If we ever want this closed, the shape is
reactive, not prophylactic: repair and retry when a guest call *fails*, which costs nothing on
the happy path and covers any reason the guest became unusable ([#38](https://github.com/conr2d/cowork/issues/38)).
Do not re-add a boot-time chmod.

### 2026-07-14 — first full pass, and an honest account of it

Every section of this document was exercised and passed, and all 15 gate-blockers found along
the way are closed. But **§0's freeze rule was not honoured**: the artifact moved eleven times
during the run, because each blocker was fixed and re-cut rather than recorded and deferred. So
sections were verified against *different* builds.

That matters here more than it usually would, because the last three fixes rewrote paths the
earlier sections stand on:

- `14a94d7` — the terminal's launch path: the shell now `exec`s the agent, so there is no shell
- `7e5650a` — the PTY teardown path: graceful HUP + grace period instead of a hard kill
- `2d26d15` — the provisioning path: the guest is written as root, not through the 9p share

§5 (sessions), §6 (continuity) and §7 (first-run sign-in) all run on top of those. They were
confirmed on `2d26d15` for the parts that were re-tested, but the run as a whole is a
composite, not a single-artifact pass.

**What was found by walking it anyway** — the reason the rule exists is throughput, not rigour,
and the run earned its keep regardless:

- The shell's terminal had **never once worked on real Windows** (#17/#19/#26). Three separate
  causes, none reproducible in CI or the dev harness.
- Guest self-heal — the mechanism I had defended in `dc912ea` on the strength of *reading* the
  code — **had never worked after provisioning** (#36). Only §9.1 could have caught it, and only
  because the gate walks a path nobody else does.
- claude's session id was being **discarded**, unrecoverably (#29); agy's conversation was never
  written at all because we hard-killed it (#34).

Three of the blockers were introduced *by* fixes made during the run (#19, #22, and the
proof-of-life flag) — which is precisely what the freeze rule exists to prevent.
